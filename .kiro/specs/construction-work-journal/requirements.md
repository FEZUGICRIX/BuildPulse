# Требования: Журнал работ на строительном объекте

## Введение

Журнал работ на строительном объекте — это функциональность внутреннего инструмента управления строительными проектами, которая позволяет отслеживать выполненные работы на объекте. Система предоставляет возможность создавать, просматривать, редактировать и удалять записи о выполненных работах с указанием даты, вида работ, объёма и исполнителя.

## Глоссарий

- **Журнал**: Система хранения и управления записями о выполненных работах
- **Запись журнала**: Единица информации о выполненной работе, содержащая дату, вид работ, объём и исполнителя
- **Вид работ**: Категория или тип выполняемой работы (например, "Кладка перегородок", "Монтаж опалубки"), включает наименование и единицу измерения
- **Объём**: Количественное значение выполненной работы
- **Единица измерения**: Мера для выражения объёма (м³, м², шт., кг и т.д.), привязана к виду работ в справочнике
- **Исполнитель**: Лицо (бригадир или рабочий), ответственное за выполнение работы
- **Справочник видов работ**: Предзаполненный список доступных видов работ
- **API**: REST API для взаимодействия фронтенда с бэкендом
- **БД**: База данных PostgreSQL для хранения данных

## Требования

### Требование 1: Просмотр списка записей журнала

**User Story:** Как руководитель объекта, я хочу просматривать список всех выполненных работ, чтобы отслеживать ход выполнения работ на объекте.

#### Acceptance Criteria

1. WHEN пользователь открывает страницу журнала, THE Журнал SHALL отобразить все записи в виде таблицы
2. THE таблица SHALL содержать следующие колонки: дата выполнения (формат ДД.ММ.ГГГГ), вид работ, объём с единицей измерения (объём с точностью до 2 знаков после запятой, единица измерения берётся из справочника видов работ), ФИО исполнителя
3. WHEN записей в журнале больше 50, THE Журнал SHALL отобразить записи с пагинацией (по 20 записей на странице) с элементами управления: кнопки "Предыдущая" и "Следующая", индикатор текущей страницы в формате "Страница X из Y"
4. THE таблица SHALL отображать записи в порядке убывания даты (новые записи сверху)
5. IF в журнале нет записей, THEN THE Журнал SHALL отобразить сообщение "Записей не найдено"
6. IF при загрузке данных произойдёт ошибка подключения к серверу, THEN THE Журнал SHALL отобразить сообщение об ошибке "Ошибка загрузки данных. Пожалуйста, проверьте соединение и обновите страницу"

### Требование 2: Фильтрация записей по дате

**User Story:** Как руководитель объекта, я хочу фильтровать записи по дате, чтобы просматривать работы за конкретный период.

#### Acceptance Criteria

1. WHEN пользователь открывает страницу журнала, THE Журнал SHALL отобразить элементы управления для фильтрации по дате
2. THE фильтр SHALL содержать два поля ввода даты: "Дата от" и "Дата до" с форматом ввода ДД.ММ.ГГГГ
3. WHEN пользователь выбирает диапазон дат и нажимает кнопку "Применить фильтр", THE Журнал SHALL отобразить только записи, дата которых попадает в выбранный диапазон (включительно с обеих сторон)
4. WHEN пользователь нажимает кнопку "Очистить фильтр", THE Журнал SHALL отобразить все записи без фильтрации и очистить поля ввода даты
5. IF диапазон дат некорректен (дата "от" позже даты "до"), THEN THE Журнал SHALL отобразить сообщение об ошибке "Дата начала не может быть позже даты окончания" и не применять фильтр
6. IF пользователь оставляет одно из полей пустым, THEN THE Журнал SHALL применить фильтр только по заполненному полю (если заполнено "Дата от", фильтровать записи с этой даты и позже; если заполнено "Дата до", фильтровать записи до этой даты включительно)

### Требование 3: Сортировка записей

**User Story:** Как руководитель объекта, я хочу сортировать записи по различным полям, чтобы быстро найти нужную информацию.

#### Acceptance Criteria

1. WHEN пользователь кликает на заголовок колонки в таблице, THE Журнал SHALL отсортировать записи по этому полю
2. THE сортировка SHALL поддерживать следующие поля: дата выполнения, вид работ, объём, ФИО исполнителя
3. WHEN пользователь кликает на заголовок колонки повторно, THE Журнал SHALL изменить направление сортировки (возрастание/убывание)
4. THE таблица SHALL отображать визуальный индикатор (стрелка) текущего направления сортировки

### Требование 4: Добавление новой записи

**User Story:** Как рабочий или бригадир, я хочу добавить запись о выполненной работе, чтобы зафиксировать выполненный объём работ.

#### Acceptance Criteria

1. WHEN пользователь нажимает кнопку "Добавить запись", THE Журнал SHALL отобразить модальное окно или форму для создания новой записи
2. THE форма SHALL содержать следующие поля: дата выполнения (date picker), вид работ (выпадающий список из справочника), объём (числовое поле), ФИО исполнителя (текстовое поле)
3. THE поля "дата выполнения", "вид работ", "объём" и "ФИО исполнителя" SHALL быть обязательными (отмечены звёздочкой или другим визуальным индикатором)
4. WHEN пользователь выбирает вид работ из выпадающего списка, THE Журнал SHALL автоматически определить единицу измерения из справочника и отобразить её рядом с полем объёма
5. WHEN пользователь заполняет форму и нажимает кнопку "Сохранить", THE Журнал SHALL выполнить валидацию на клиенте перед отправкой на сервер
6. IF какое-либо обязательное поле не заполнено, THEN THE Журнал SHALL отобразить сообщение об ошибке рядом с соответствующим полем (например, "Это поле обязательно") и не отправлять данные на сервер
7. WHEN все обязательные поля заполнены корректно, THE Журнал SHALL отправить данные на сервер через API POST /api/work-logs с телом: { date, workTypeId, volume, executorName }
8. WHEN сервер успешно сохранит запись (HTTP 201), THE Журнал SHALL закрыть форму, отобразить сообщение об успехе "Запись успешно добавлена" и обновить список записей
9. IF при сохранении произойдёт ошибка сервера (HTTP 400, 500), THEN THE Журнал SHALL отобразить сообщение об ошибке пользователю и оставить форму открытой для повторного редактирования

### Требование 5: Валидация данных при добавлении записи

**User Story:** Как система, я хочу валидировать данные при добавлении записи, чтобы обеспечить целостность данных в журнале.

#### Acceptance Criteria

1. WHEN пользователь вводит дату выполнения, THE Журнал SHALL проверить, что дата не позже текущей даты
2. IF дата позже текущей даты, THEN THE Журнал SHALL отобразить сообщение об ошибке "Дата не может быть в будущем"
3. WHEN пользователь вводит объём, THE Журнал SHALL проверить, что значение является положительным числом
4. IF объём меньше или равен нулю, THEN THE Журнал SHALL отобразить сообщение об ошибке "Объём должен быть больше нуля"
5. WHEN пользователь вводит ФИО исполнителя, THE Журнал SHALL проверить, что поле содержит от 3 до 100 символов
6. IF ФИО содержит менее 3 или более 100 символов, THEN THE Журнал SHALL отобразить сообщение об ошибке "ФИО должно содержать от 3 до 100 символов"

### Требование 6: Редактирование существующей записи

**User Story:** Как руководитель объекта, я хочу редактировать существующую запись, чтобы исправить ошибки или обновить информацию.

#### Acceptance Criteria

1. WHEN пользователь кликает на кнопку "Редактировать" рядом с записью, THE Журнал SHALL отобразить форму редактирования с текущими данными записи
2. THE форма редактирования SHALL содержать те же поля, что и форма добавления
3. WHEN пользователь изменяет данные и нажимает кнопку "Сохранить", THE Журнал SHALL валидировать все обязательные поля
4. WHEN все данные корректны, THE Журнал SHALL отправить обновленные данные на сервер через API
5. WHEN сервер успешно обновит запись, THE Журнал SHALL закрыть форму и обновить список записей
6. IF при обновлении произойдёт ошибка сервера, THEN THE Журнал SHALL отобразить сообщение об ошибке пользователю

### Требование 7: Удаление записи

**User Story:** Как руководитель объекта, я хочу удалить ошибочную запись, чтобы исправить данные в журнале.

#### Acceptance Criteria

1. WHEN пользователь кликает на кнопку "Удалить" рядом с записью, THE Журнал SHALL отобразить диалог подтверждения
2. THE диалог подтверждения SHALL содержать сообщение "Вы уверены, что хотите удалить эту запись? Это действие нельзя отменить."
3. WHEN пользователь нажимает кнопку "Удалить" в диалоге, THE Журнал SHALL отправить запрос на удаление на сервер через API
4. WHEN сервер успешно удалит запись, THE Журнал SHALL закрыть диалог и обновить список записей
5. IF при удалении произойдёт ошибка сервера, THEN THE Журнал SHALL отобразить сообщение об ошибке пользователю
6. WHEN пользователь нажимает кнопку "Отмена" в диалоге, THE Журнал SHALL закрыть диалог без удаления записи

### Требование 8: Справочник видов работ

**User Story:** Как система, я хочу предоставить справочник видов работ, чтобы обеспечить единообразие данных и упростить ввод информации.

#### Acceptance Criteria

1. WHEN пользователь открывает форму добавления или редактирования записи, THE Журнал SHALL отобразить поле "Вид работ" как выпадающий список
2. THE выпадающий список SHALL загружать данные из API GET /api/work-types
3. THE каждый вид работ в справочнике SHALL содержать: id, наименование (name) и единицу измерения (unit)
4. THE выпадающий список SHALL отображать наименование вида работ и единицу измерения в формате "Наименование (единица)"
5. WHEN пользователь выбирает вид работ из списка, THE Журнал SHALL сохранить workTypeId для отправки на сервер
6. THE справочник SHALL быть предзаполнен следующими видами работ: "Кладка перегородок (м³)", "Монтаж опалубки (м²)", "Бетонирование (м³)", "Монтаж арматуры (кг)", "Отделочные работы (м²)", "Прочие работы (шт)"

### Требование 9: REST API для управления записями

**User Story:** Как фронтенд приложение, я хочу взаимодействовать с бэкендом через REST API, чтобы выполнять операции CRUD над записями журнала.

#### Acceptance Criteria

1. THE API SHALL предоставить endpoint GET /api/work-logs для получения списка всех записей с поддержкой фильтрации, сортировки и пагинации
2. THE API SHALL поддерживать параметры фильтрации: dateFrom (ISO 8601 формат YYYY-MM-DD), dateTo (ISO 8601 формат YYYY-MM-DD)
3. THE API SHALL поддерживать параметры сортировки: sortField (допустимые значения: date, workType, volume, executorName), sortOrder (asc или desc)
4. THE API SHALL поддерживать параметры пагинации: page (начиная с 1), limit (максимум 100, по умолчанию 20)
5. THE GET /api/work-logs SHALL возвращать объект с полями: data (массив записей с вложенным объектом workType), total (общее количество записей), page (текущая страница), limit (размер страницы)
6. THE API SHALL предоставить endpoint GET /api/work-types для получения справочника видов работ
7. THE API SHALL предоставить endpoint POST /api/work-logs для создания новой записи с DTO: { date: string (YYYY-MM-DD), workTypeId: string (UUID), volume: number, executorName: string }
8. THE POST /api/work-logs SHALL валидировать DTO через class-validator с правилами: date обязателен и не позже текущей даты, workTypeId обязателен и существует в таблице work_types, volume обязателен и > 0, executorName обязателен и содержит от 3 до 100 символов
9. THE POST /api/work-logs SHALL возвращать HTTP 201 Created с созданной записью (включая вложенный объект workType) при успехе
10. THE API SHALL предоставить endpoint PATCH /api/work-logs/:id для обновления существующей записи с тем же DTO, что и POST
11. THE PATCH /api/work-logs/:id SHALL валидировать DTO с теми же правилами, что и POST
12. THE PATCH /api/work-logs/:id SHALL возвращать HTTP 200 OK с обновленной записью (включая вложенный объект workType) при успехе
13. THE API SHALL предоставить endpoint DELETE /api/work-logs/:id для удаления записи
14. THE DELETE /api/work-logs/:id SHALL возвращать HTTP 204 No Content при успехе
15. IF запись не найдена (GET, PATCH, DELETE), THEN THE API SHALL возвращать HTTP 404 Not Found с сообщением об ошибке
16. IF данные в DTO некорректны (POST, PATCH), THEN THE API SHALL возвращать HTTP 400 Bad Request с описанием ошибок валидации для каждого поля в формате: { message: string, errors: { field: string[] } }
17. IF workTypeId не существует в таблице work_types (POST, PATCH), THEN THE API SHALL возвращать HTTP 400 Bad Request с сообщением "Вид работ не найден"
18. IF произойдёт внутренняя ошибка сервера, THEN THE API SHALL возвращать HTTP 500 Internal Server Error с сообщением об ошибке

### Требование 10: Схема базы данных

**User Story:** Как система, я хочу хранить данные о записях журнала в БД, чтобы обеспечить персистентность и надёжность данных.

#### Acceptance Criteria

1. THE БД SHALL содержать таблицу work_types (справочник видов работ) с колонками: id (UUID, первичный ключ), name (VARCHAR(255), уникальный, не NULL), unit (VARCHAR(50), не NULL), createdAt (TIMESTAMP), updatedAt (TIMESTAMP)
2. THE БД SHALL содержать таблицу work_logs (записи журнала) с колонками: id (UUID, первичный ключ), date (DATE, не NULL), workTypeId (UUID, внешний ключ на work_types.id, не NULL), volume (DECIMAL(10,2), не NULL), executorName (VARCHAR(100), не NULL), createdAt (TIMESTAMP), updatedAt (TIMESTAMP)
3. THE колонка id SHALL генерироваться автоматически (UUID v4)
4. THE колонка createdAt SHALL устанавливаться автоматически при создании записи (CURRENT_TIMESTAMP)
5. THE колонка updatedAt SHALL обновляться автоматически при изменении записи (CURRENT_TIMESTAMP)
6. THE внешний ключ workTypeId SHALL обеспечивать ссылочную целостность (ON DELETE RESTRICT)
7. THE таблица work_types SHALL быть предзаполнена (seed) следующими видами работ: "Кладка перегородок" (м³), "Монтаж опалубки" (м²), "Бетонирование" (м³), "Монтаж арматуры" (кг), "Отделочные работы" (м²), "Прочие работы" (шт)

### Требование 11: Обработка ошибок на фронтенде

**User Story:** Как пользователь, я хочу видеть понятные сообщения об ошибках, чтобы понять, что произошло и как исправить проблему.

#### Acceptance Criteria

1. IF сервер недоступен, THEN THE Журнал SHALL отобразить сообщение об ошибке "Ошибка подключения к серверу. Пожалуйста, проверьте интернет-соединение и попробуйте снова"
2. IF сервер возвращает ошибку валидации (HTTP 400), THEN THE Журнал SHALL отобразить сообщение об ошибке с описанием проблемы
3. IF сервер возвращает ошибку 404, THEN THE Журнал SHALL отобразить сообщение об ошибке "Запись не найдена"
4. IF сервер возвращает ошибку 500, THEN THE Журнал SHALL отобразить сообщение об ошибке "Ошибка сервера. Пожалуйста, попробуйте позже"

### Требование 12: Управление состоянием формы на фронтенде

**User Story:** Как фронтенд приложение, я хочу правильно управлять состоянием формы, чтобы обеспечить хороший пользовательский опыт.

#### Acceptance Criteria

1. WHEN пользователь заполняет форму, THE Журнал SHALL отслеживать изменения в каждом поле
2. WHEN пользователь нажимает кнопку "Сохранить", THE Журнал SHALL отключить кнопку и отобразить индикатор загрузки
3. WHEN запрос на сервер завершится, THE Журнал SHALL включить кнопку и скрыть индикатор загрузки
4. WHEN пользователь нажимает кнопку "Отмена", THE Журнал SHALL закрыть форму без сохранения изменений
5. WHEN пользователь закрывает форму с несохраненными изменениями, THE Журнал SHALL отобразить диалог подтверждения "У вас есть несохраненные изменения. Вы уверены, что хотите закрыть форму?"
